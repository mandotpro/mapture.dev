# Task 001.2: Git Hooks and Strict Code Quality (Linting & Tests)

## Goal
Establish an automated code quality gauntlet locally on developers' machines (and agents) using standard Git pre-commit and pre-push hooks. This will enforce a strict Go structural styling and ensure no regressions occur before code hits version control.

## Context
Before writing extensive backend CLI and parser logic, enforcing standardized linting (`golangci-lint`) and robust testing prevents spaghetti code, messy PRs, and unnoticed regressions. Additionally, the hooks ensure that our isolated `./examples/` directory formally bridges with our compiled tests to run realistic integration assertions.

## Requirements

### 1. `golangci-lint` Configuration
- Define an aggressive baseline `.golangci.yml` strictly targeting the logic inside the `./src/` path.
- Enable basic stylistic checks, cyclomatic complexity checkers, and rigorous error-handling verifiers.

### 2. Pre-Commit Hook (`.git/hooks/pre-commit`)
- The script should enforce standard formatting by running `go fmt ./src/...`.
- Run `golangci-lint run ./src/...`.
- The hook must block `git commit` structurally if these jobs exit with non-zero codes (i.e. if code is messily formatted, or if lint fails).

### 3. Pre-Push Hook (`.git/hooks/pre-push`)
- The script should run the entire Go test suite: `go test -v ./src/...`.
- The Go tests implemented throughout the repository MUST load scenarios and dummy cases natively using files inside `./examples/` (like `examples/demo/architecture/`). Thus, `pre-push` natively exercises our actual example structures to guarantee the tool works end-to-end.

### 4. Hook Setup Automation (`scripts/init-hooks.sh`)
- Git hooks aren't tracked natively inside `.git` by default on clone. Provide a minimalist `Makefile` or an `init-hooks.sh` script inside a `scripts/` directory that developers can run exactly once to securely copy the hooks into their local `.git/hooks/` registry.

## Definition of Done
- A developer can run `./scripts/init-hooks.sh` and receive functioning pre-commit and pre-push triggers.
- Attempting to commit unformatted Go code automatically formats it or blocks the commit.
- Attempting to push failing tests (broken by a change to `/src/` behavior over an `/examples/` schema payload) rejects the push operation seamlessly locally.
